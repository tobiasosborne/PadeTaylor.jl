## Design synthesis

The figure script and tests both work in two layers:

### Layer 1: sheet-aware asymptotic IC wrapper (figure-local)

The shipped `pIII_asymptotic_ic(z; ...)` uses Julia's principal
complex `z^{1/3}`.  For FFW md:243's upper IC at `arg z = 13π/6`,
the principal cube root differs from the sector-correct branch by
a phase of `2π/3` in `s = z^{1/3}` — gives completely different
`u`.  The figure script defines `tronquee_ic_sheet(z_modulus, arg_z;
n_terms, β, δ)` that takes `arg z` (the unfolded argument) as a
separate parameter, builds `s = |z|^{1/3} · exp(i · arg_z / 3)`
with the caller's choice of branch, then sums the series
`u = s · (1 + a_1 s^{-2} + a_2 s^{-4})`.  The coefficients `a_1 =
-β/3`, `a_2 = ((4/9) + δ a_1²) / (2δ)` are lifted verbatim from
the shipped helper's docstring.

This wrapper is figure-local — the shipped helper is byte-unchanged
(CLAUDE.md Rule 6 + bead spec mandate).

### Layer 2: principal-sheet sub-strip render

The full FFW sector spans 6π in `Im ζ`.  Sheet-aware asymptotic
`u(z)` evaluation across this 1.5-sheet range requires routing
`ζ` (not just `z`) through the hybrid-driver API — a refactor of
`IVPBVPHybrid.solve_pole_free_hybrid` we explicitly **do not**
do at this step (bead constraint).

The honest v1 deliverable: render the PRINCIPAL-sheet sub-strip
`Im ζ ∈ [-3π/2 + 0.05, 2π - 0.05]` (≈ 3.47π wide).  This covers
the FFW lower-IC at `arg z = -2π/3` (`Im ζ = -4π/3 ≈ -4.19`, well
inside) and the FFW pole-free sector boundary at `arg z = 9π/4`
(`Im ζ = 9π/2`, just outside the rendered sub-strip — the upper
sub-strip boundary `Im ζ = 2π - 0.05` is at `arg z = π - 0.025`).

Panel B (closed-form κ heatmap) renders over the FULL FFW sector
because the formula is `Im ζ`-independent.  Panel B's annotated
`ζ₁ FFW` cross sits at `Im ζ = 13π/3 ≈ 13.61`, in the upper sheet
beyond Panel A's rendered range — explicitly the v1 corner.

### Layer 3: the call

```julia
sector = (im_lo = -3π/2 + 0.05, im_hi = 2π - 0.05,
           re_anchor = 2 log 30, re_extent = 3.0)
ic_fn  = make_principal_sheet_ic(sector; n_terms = 15, β = -1/20, δ = -1)
sol    = solve_pole_free_hybrid(pp, sector, ic_fn;
            pfs_kwargs = (; h = 0.4, adaptive_tol = 1e-10, ...),
            bvp_kwargs = (; N = 50, tol = 1e-10, maxiter = 30),
            n_slices = 12, glue_tol = 1e-8)
```

All 12 BVP slices converge (final residual ≤ 5e-9; iteration counts
from 3 to 23).  Panel A samples `sol(ζ)` over a 220×320 grid;
100% finite coverage.

### Headline findings

1. **`n_terms` has NO effect** beyond `n_terms = 2` at v1.  The
   shipped `pIII_asymptotic_ic` hard-codes `a_3 ... a_N = 0`
   (worklog 039 §"What is NOT shipped" + ADR-0014 §"Out of scope").
   So `n_terms = 5, 10, 15, 20` all return the same value as
   `n_terms = 2`.  The bead spec's instruction "push n_terms ≥ 15"
   is therefore a no-op at v1 — it's what the v2 lift would enable
   if the helper grew higher-order coefficients (the deferred bead
   `padetaylor-ykg`).

2. **Achievable v1 |Δu| floor ≈ 10⁻²** at the SECTOR INTERIOR (vs
   the asymptotic series's own evaluation).  Breakdown:
     * Asymptotic-series truncation at `n_terms = 2`: dominated by
       the `a_3 z^{-7/3}` contribution which the v1 helper drops.
       At |z| = 30, `|z^{-7/3}| ≈ 3e-4`, with `a_3 ~ O(1)` this
       contributes ~3e-4 in u.  This is the FUNDAMENTAL floor at v1.
     * BVP spectral floor at `N = 50`: ~1e-12 (well below the
       series error).
     * Numerical-controller noise in the PFS: ~1e-10.
     * Sum: dominated by the series error; |Δu| ≈ 1e-2 observed.

3. **High N (70) breaks Newton** — multi-basin issue from worklog
   039 §"Friction 2".  Different N values land Newton in different
   basins; at high N the BVP discretization gains spurious local
   minima.  The figure uses `N = 50` (stable, midrange) per the
   bead spec.

4. **FFW md:240 per-strip errors `4e-6 → 4e-8` are UNREACHABLE at v1**.
   The gap is ~4 orders of magnitude.  Path to closing it:
     * v2a: implement `a_3, a_4, ...` in `pIII_asymptotic_ic` (the
       deferred refinement, ~150 LOC of TaylorSeries.jl substitution)
       — would buy ~5 orders in series truncation accuracy.
     * v2b: BF-256 precision pipeline — would buy precision floor
       from `eps(Float64) ≈ 2e-16` to `eps(BF256) ≈ 10⁻⁷⁷`, freeing
       the FFW md:240 numbers if v2a also lands.
     * v2c: 2D Chebyshev spectral BVP on the sector (replace 1D
       slice stack + linear interp) — would tighten the BVP-self
       precision floor.
   Each is a separate bead (`padetaylor-ykg`, `padetaylor-7zw`-style,
   `padetaylor-?2d` (a 2D-BVP follow-up, to be filed when relevant)).

## Friction log

### 1.  The sheet-aware cube-root rabbit hole.

First attempt: build a closure `make_sheet_aware_ic(sector)` that
recovers the unfolded `arg z` from Julia's principal `angle(z)` via
an integer winding `k` chosen to place `arg_principal + 2π k` inside
the sector's `Im ζ / 2` range.  This works for sectors of width ≤ 2π
in `arg z` (≤ 4π in `Im ζ`), where the integer `k` is unique.  FFW
Fig 5 is 3π wide in `arg z` — two valid `k`'s near the centre, and
the heuristic (pick the closest-to-midpoint) gives the WRONG sheet
for queries at the sector extremities.

Failed: the driver's `ζ_top = re_anchor + i (im_hi - ε)` at
`Im ζ ≈ 9π/2 - ε` has `arg z = (9π/2 - ε)/2 ≈ 9π/4`.  Julia's
`angle(exp(ζ_top/2)) = (Im ζ_top / 2) mod 2π = (9π/4) mod 2π = π/4`.
My midpoint heuristic chose `k = 0` (giving `arg = π/4`), but the
sector-correct choice is `k = 1` (giving `arg = 9π/4`).  The cube
root then computes the WRONG sheet of `u = z^{1/3}` and the PFS
walks anchored there transport garbage along the boundary.

Resolution: rather than build a correct sheet-aware closure (which
needs `ζ` not `z`, requiring an API change), **restrict the v1
render to the principal-sheet sub-strip** where Julia's principal
`z^{1/3}` is correct.  Panel B (closed-form κ) extends across the
full FFW sector because the formula is Im ζ-independent.

This is **CLAUDE.md Rule 9** discipline: "Senior-engineer-grade only.
... If a v1-acceptable corner exists, document it as a deferred bead
with the exact condition that would force v2 work."  The condition:
"future B5 / B6 needing the full multi-sheet sector → file
`padetaylor-ykg` for `IVPBVPHybrid` API extension allowing `ζ` to
flow through to the asymptotic-IC closure."

### 2.  `n_terms = 15` is a no-op at v1 (the bead spec's silent constraint).

The bead spec says "Push n_terms ≥ 15 in asymptotic series".  Worklog
039 §"What is NOT shipped" already documents that the shipped helper
hard-codes `a_3 ... a_N = 0`.  Cranking n_terms therefore changes
nothing — the bead spec's directive can only be honored after a v2
helper lift.

I considered porting FFW md:232 "optimal truncation" (TaylorSeries.jl
substitution of the `s = z^{1/3}` Laurent series into the canonical
PIII equation, solving the resulting upper-triangular linear system
for `a_3, a_4, ..., a_15`).  Estimated ~150 LOC.  Scope creep — the
bead is "render Fig 5 with the hybrid driver" not "lift the
asymptotic-series helper".  Deferred to bead `padetaylor-ykg`.

The figure script uses `n_terms = 15` as a *future-compatible*
choice — when the v2 helper lands, the figure script auto-tightens
without source changes.

### 3.  BVP Newton convergence at high N.

Sweep 2 row 5 (`n_terms = 20, N = 70`) fails: Newton diverges with
`||Δu||_∞ > 60`.  Worklog 039 §"Friction 2" already documents this:
the BVP at `N ∈ {20, 24, 30}` lands in alternate basins (yielding
*different* tronquée or non-tronquée solutions to the discretization).
At `N = 50` (the figure's choice) the basin is consistent; at `N = 70`
on the 6π-wide sector it's not.

Pragmatic v1 response: use `N = 50` everywhere.  The deferred lift
(`padetaylor-?2d` (a 2D-BVP follow-up, to be filed when relevant)) is a 2D spectral BVP that doesn't have the
slice-Re-direction-dependence pathology.

### 4.  The FFW sector is sheet-bridging, not just 1.5 sheets.

I initially read FFW md:222 — "pole-free on `-3π/4 < arg z < 9π/4`"
— as a single sector on a single (multi-valued) z-Riemann surface
sheet.  Actually it's 1.5 SHEETS of the PIII Riemann surface bridged
by the pole-free condition (PIII has branch point at z = 0 with
sheet-width = 2π in arg z, so 3π/2 + 9π/4 ÷ 2π ≈ 1.5 sheets).  The
ζ-frame view (`Im ζ ∈ (-3π/2, 9π/2)`, 6π wide = 1.5 × 4π) makes the
multi-sheet structure visible: ANY  asymptotic-IC defined on `z` must
specify which `z^{1/3}` branch is used.  This is the underlying
reason §"Friction 1" arose.

The principal-sheet sub-strip rendered for Panel A covers `Im ζ ∈
[-3π/2 + 0.05, 2π - 0.05]` — i.e. the LOWER 0.875 sheet of the FFW
sector.  Re-rendering the UPPER 0.875 sheet (or the FULL sector)
needs the sheet-aware machinery.

## Hard-won lesson

**The hybrid driver's API contract — `asymptotic_ic_fn(z)` returning
`(u, u')` — is a 1-sheet API.**  ADR-0014's two-fold IC anchoring
(`ζ_top`, `ζ_bot` in `solve_pole_free_hybrid`) assumes both anchors
share the same `z^{1/3}` branch by virtue of being near each other
in `z`.  For FFW Fig 5's 1.5-sheet sector that's geometrically
impossible: the two anchors live on DIFFERENT z-Riemann sheets.

The right v2 generalization is to lift the API to take `ζ` directly:

```julia
asymptotic_ic_fn(ζ::Complex) -> (w, w')   # ζ-frame, sheet-explicit
```

This is a **breaking API change** to `solve_pole_free_hybrid` — not
in this bead's scope.  The v1 honest workaround (principal-sheet
sub-strip render) catches what's available without forcing the
refactor; the figure's Panel B at full sector + the explicit
sub-strip-callout in Panel A's caption keeps the FFW context visible.

This is the same "compose, don't refactor" pattern as worklog 039's
"hybrid drivers are exhibitions of their components' regimes, not
refinements of them" — except here the *bead boundary* itself was
the constraint that surfaced the API limitation.
