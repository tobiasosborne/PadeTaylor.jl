## Ground truth

- `references/markdown/FW2011_painleve_methodology_JCP230/FW2011_painleve_methodology_JCP230.md:139-145`
  — Fig 3.1: `|u(z)|` for PI `u'' = 6u² + z` with the near-tritronquée
  ICs `u(0) = -0.1875`, `u'(0) = 0.3049`, over `-10 ≤ x,y ≤ 10`. The
  ICs are *close to but not equal to* the tritronquée values, so
  there are pole fields in all five main sectors, four of them
  displaced from the origin.
- `...md:151-164` — Fig 3.2: the FW 2011 §3.1 two-stage path network.
  Stage 1 grows a tree of length-`h` steps from the origin that
  reaches within `h` of every node of a coarse `40×40` grid; each
  step picks, among five wedge directions (straight at goal, ±22.5°,

  the nearest already-visited node, so the paths form a single
  non-crossing tree. FW uses `h = 0.3` for this illustration and
  reports "about 1540 steps were needed for 1600 target points".

## What changed

### `src/PathNetwork.jl` — `visited_parent` tree tracking

`PathNetworkSolution` gained a sixth `visited_*` field:

```julia
visited_parent :: Vector{Int}    # tree edge: parent index, 0 = root
```

`visited_parent[1] = 0` (the IC/root node has no parent). Inside the
Stage-1 walk loop, a `parent_idx` cursor is threaded: it starts at
`idx_v` (the nearest already-visited node the per-target walk begins
from — FW 2011 line 164) and, after each `push!`, advances to the
just-pushed node's index, so subsequent steps of the same walk chain
off their immediate predecessor. Both constructor call sites updated
(the main driver and `_solve_with_schwarz_reflection`, which forwards
the upper-half walk's `visited_parent` verbatim — the visited tree is
the upper walk's; only Stage-2 grid outputs are mirrored). Module +
struct docstrings updated to describe the field and the edge set.

No other module constructs `PathNetworkSolution` positionally
(`LatticeDispatcher`, `Dispatcher`, `SheetTracker` only *hold* or
reference the type), so appending the field is non-breaking.

### `test/pathnetwork_test.jl` — `PN.5.1`

New testset pins the structural invariants that make `visited_parent`
a well-formed tree:

  - one parent slot per visited node;
  - root parent is `0`;
  - every non-root parent precedes its child (`1 ≤ p ≤ k-1`) — no
    cycles, incremental construction;
  - **every tree edge has Euclidean length ≈ h** — each visited node
    is reached by exactly one wedge step (length `h`) from its
    parent, the first node of a per-target walk included;
  - following parent pointers from any node reaches the root.

**Mutation-proof** (verified): deleting the
`parent_idx = length(visited_z)` line in `src/PathNetwork.jl` makes
every node of a multi-step walk point at the walk's *start* node
instead of its immediate predecessor; invariant "edge length ≈ h"
then goes RED for every node ≥ 2 hops into a walk (its edge to the
start node has length `k·h`, not `h`). Restored.

### `figures/` — new figure-reproduction project

A self-contained Julia project (`figures/Project.toml`) depending on
`CairoMakie` + a `dev`-ed `PadeTaylor`, so the heavyweight plotting
dependency never enters the package's own `Project.toml`. One script
per figure; each is literate exposition of the source figure, the
ODE/ICs, and the grid parameters. `figures/Manifest.toml` is
gitignored; the output PNGs are committed (they are the deliverable).

  - `figures/fw2011_fig_3_1.jl` — solves PI on the fine `N×N` lattice
    (`N = 121`; FW used 161) with `path_network_solve`,
    `enforce_real_axis_symmetry = true` (PI has real coefficients and
    the ICs are on the real axis, so `u(z̄) = ū(z)` exactly — halves
    the work). `|u(z)|` is capped at 50 and drawn as an `Axis3`
    surface, matching FW's Fig 3.1 framing.
  - `figures/fw2011_fig_3_2.jl` — grows the Stage-1 tree on the exact
    `40×40` coarse grid at `h = 0.3` (plain full-plane algorithm, not
    the Schwarz shortcut — Fig 3.2 illustrates the actual asymmetric
    tree the paper's algorithm produces), and draws the coarse grid
    as dots, the `visited_parent` edge set as line segments, and the
    origin as a solid black circle.

## Empirical results

### Fig 3.2 — path tree

`40×40` grid, `h = 0.3`, `order = 30`, ran in **4.8 s**. Produced a
**2471-step** tree for the 1600 target points (FW: "about 1540").
The visual topology is a clear match for FW's `_page_4_Figure_9.jpeg`:
a single connected tree rooted at the origin, paths curving through
the low-`|u|` passages between pole fields, no crossings.

The step count runs ~1.6× FW's. This is within the latitude of FW's
own "about" — the count is sensitive to (a) the random target order
(a different RNG visits targets in a different sequence, producing a
different tree), and (b) how aggressively `:min_u` detours around
PI's pole fields with these particular near-tritronquée ICs. It is
not a correctness signal — `PN.5.1` confirms the tree is well-formed,
and the topology matches. Worth a future probe if we want to match
FW's count more tightly (the wedge tie-break rule and the
exact-`h`-vs-`≤h` halt condition are the likely levers), but not a
blocker for the visual-match acceptance Fig 3.2 carries.

### Fig 3.1 — pole-field surface

`121×121` lattice over `[-10,10]²`, `h = 0.5`, `order = 30`,
`enforce_real_axis_symmetry = true`, ran in **4.2 s** (907 visited
tree nodes; all 14641 lattice points covered, no `NaN` sentinels).
The `Axis3` surface + faint wireframe reproduces FW's
`_page_3_Figure_8.jpeg`: a lattice of pole spikes (capped at `|u| =
50`) over a comparatively calm central patch.

Quantitative companion to the visual match (the script prints it):
the **median `|u|` over the central disc `|z| ≤ 2` is `0.491`, vs
`1.538` over the outer annulus `6 ≤ |z| ≤ 10` — a 3.1× ratio**. The
calm-near-origin / pole-fields-displaced-outward signature FW
describes (md:139) is therefore present numerically, not just to the
eye. This is the visual companion to the already-shipped Phase 9
quantitative pin (`test/phase9_tritronquee_test.jl`, worklog 012),
which verified 4-of-5 pole-free sectors at `25×25` over `[-4,4]²`.

## Frictions

  - **`PathNetworkSolution` had nodes but no edges.** Phase 10 stored
    the visited-node list but never recorded which node each was
    stepped from — fine for Stage-2 evaluation (nearest-visited
    lookup is geometric), but it meant the Stage-1 *tree* could not
    be reconstructed unambiguously after the fact (a node could be
    one wedge step from several earlier nodes). Drawing FW Fig 3.2
    faithfully forced the field to exist. Lesson in the same spirit
    as hard-won lesson 12: a data structure that an ADR describes as
    "a tree" should actually store its edges.
