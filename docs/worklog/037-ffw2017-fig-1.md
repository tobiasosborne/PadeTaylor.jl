## Ground truth read first

Per Law 1, foundational reads were FFW 2017 §2.1 + the Fig 1 caption
+ Table 2:

  - **md:98-105** — Figure 1 caption.  Parameters
    `(α, β, γ, δ) = (-1/2, -1/2, 1, -1)`, IC
    `(u(1), u'(1)) = (1/4, 1)` in the z-plane (md:101).  Node
    separation `R(ζ) = (8 - Re ζ)/20` (md:72/101).  Three strips
    `Im ζ ∈ (-2π + 4π s, 2π + 4π s]` for `s ∈ {-1, 0, 1}` (md:103).
    Stage-2 fine grid `571 × 140` with spacing `1/15` (md:107).
    **2701 visited tree nodes** (md:95) / **3041 Stage-1 targets**
    (md:95).

  - **md:41-43** — PIII exponential transform `z = exp(ζ/2)`,
    `u(z) = exp(-ζ/2) w(ζ)` and the modified P̃_III equation
    (different from PV: note the `e^{-ζ/2}` rescale on `w`).
    Strip width **4π** in ζ corresponds to ONE sheet in z (vs PV's
    2π strip — the factor of 1/2 in the transform).

  - **md:122** — FFW's error estimator
    `E(ζ) = |w(ζ) - conj(w(conj ζ))| / |w(ζ)|`, exploiting the
    conjugate symmetry that holds for real parameters + real IC.
    We adopt this verbatim as the sheet-0 oracle in FF1.1.4.

  - **md:128-133** — Table 2.  Two experiments.  Experiment 1
    (Tol = 1e-14): adaptive `4e-10 / 4e-8` sheets 0 / ±1.
    Experiment 2 (Tol = 1e-11): adaptive `1e-6 / 8e-2` sheets 0 / ±1.
    Our figure runs at `Tol = 1e-10` — one order looser than
    Experiment 2.

  - **md:72** — "pole density will increase rapidly on the region
    Re ζ ≫ 0".  Pinned quantitatively by FF1.1.6.

  - **md:105** — "spirals of poles whirling around the branch point
    z = 0 in the fourth column is a pole field pattern that has
    not been observed before".  This is the visual point of Fig 1.

### Rendering decision (vs B1 / Fig 6 — the headline of this worklog)

The bead spec called out that B1's `figures/ffw2017_fig_6.jl`
z-plane rendering uses `scatter!` on polar-projected lattice
points (`(r cos θ, r sin θ)` markers), producing visible spoke
artefacts — rays of dots radiating from the origin where the
polar mesh becomes coarse at small `r`.  FFW's actual figures
use smooth filled surfaces.

The B2 upgrade: **`heatmap!` on a uniform Cartesian z-grid via
bilinear interpolation from the ζ-grid**.

```julia
function bilinear_w(ζ::Complex)
    rx, iy_ = real(ζ), imag(ζ)
    (rx < xs[1] || rx > xs[end] || iy_ < ys[1] || iy_ > ys[end]) &&
        return complex(NaN, NaN)
    dx = xs[2] - xs[1]; dy = ys[2] - ys[1]
    i = clamp(floor(Int, (rx - xs[1]) / dx) + 1, 1, NX - 1)
    j = clamp(floor(Int, (iy_ - ys[1]) / dy) + 1, 1, NY - 1)
    tx = (rx - xs[i]) / dx
    ty = (iy_ - ys[j]) / dy
    w00 = W[i, j]; w10 = W[i+1, j]; w01 = W[i, j+1]; w11 = W[i+1, j+1]
    (all isfinite) || return complex(NaN, NaN)
    return (1-tx)(1-ty) w00 + tx(1-ty) w10 + (1-tx)ty w01 + tx·ty·w11
end
```

For each Cartesian z-cell on sheet `s`:

  1. Compute `ζ = 2(log|z| + i arg z) + 4π i s` (sheet selection
     via the `4π i s` shift on top of the principal-branch `log`).
  2. Bilinear-interpolate `w(ζ)` from the regular ζ-grid `W[i, j]`
     at axes `xs`, `ys`.
  3. Apply the inverse transform `u = w / z`, so `|u(z)| = |w|/|z|`.
  4. NaN out cells where ζ falls outside the walked window or any
     interpolant corner is non-finite; render `nan_color = :transparent`.

The result is a smooth modulus surface in the z-plane that matches
FFW's depiction.  No spoke artefacts.  The trade-off: ~16% of
Cartesian z-cells fall outside the visited tree's reach (mostly
at the strip boundaries `Im ζ ≈ ±6π` where the walker stops
short).  These render as transparent gaps; for a publication
figure they could be filled by extending the ζ-window or by
masking with the convex hull of the visited tree.

The Cartesian-resample approach is what FFW themselves implicitly
do — their Stage-2 fine grid (FFW md:107) is uniform in `ζ`, then
the modulus is plotted at `(r, θ) = (exp(Re ζ / 2), Im ζ)`.  Same
underlying data, different rendering convention.  B1 used Makie's
`scatter!` on the polar mesh directly (~21,600 markers at
`markersize = 2`), which makes spokes visible.  B2's Cartesian
resample erases that artefact.

### Mutation-proof bites (verified, footer-documented)

  - **M1** — swap α ↔ γ to `(1, -1/2, -1/2, -1)`.  The walker
    **THROWS** at `Re ζ ≈ 1.2, Im ζ ≈ 10`: "all 5 wedge candidates
    failed".  The mutated PIII has a different pole geometry the
    walker cannot navigate from the FFW IC.  Strongest possible
    bite (test setup itself fails before assertions run).

  - **M2** — replace `R(ζ) = max(0.1, (8 - Re ζ)/20)` with
    constant `R(ζ) = 0.5`.  Walker **THROWS** at
    `Re ζ ≈ 4.4, Im ζ ≈ -11.7`.  Without spatial-density tracking,
    the walker steps over poles at the constant `h = 0.5` rather
    than threading between them at `h = 0.1`.  ADR-0012 alternative
    D (skip R, rely on adaptive alone) is empirically rejected by
    this exact failure mode.

  - **M3** — `step_size_policy = :fixed` with `h = 0.5`, drop
    `node_separation`.  Walker COMPLETES (the FW-2011 fixed-h
    regime is robust on real-axis-symmetric problems per FW Table
    5.1) but produces a much sparser visited tree (~820 nodes vs
    baseline ~2664).  Bites **FF1.1.2**'s visited-count floor of
    2296.  The absolute pin at `0.5 + 1.0i` does NOT bite M3
    (that point is close enough to the IC that even a fixed-h
    walker reaches it accurately) and the pole-density gradient
    does NOT bite (the visited tree, though sparse, still tracks
    pole density via random target ordering).  FF1.1.2 is the
    load-bearing M3 detector.

All three mutations bite at least one FF1.* assertion at a
distinct assertion class.  Test catches them.

## Frictions

### The pole-density wall at `Re ζ ≳ 5.5`

The first probe ran the walker over FFW's full `Re ζ ∈ [-2, 8]`
window.  At every tolerance setting tried (`1e-10`, `1e-12`) the
walker threw "all 5 wedge candidates failed" somewhere in the
high-Re-ζ region.  Initial misdiagnosis: "too dense a target
grid".  Reducing density helped a little but didn't fix it.  The
**root cause** (per Rule 2 "All bugs are deep") is FFW's published
limitation: their Poisson-disk node placement (ref [8],
`Fornberg 2015`) tracks the *actual* pole density rather than
rectangular grid sampling, and is essential to navigate
`Re ζ ≳ 5` reliably.

Pulling back to `Re ζ ≤ 5` gave us the FFW node counts within
±15% AND eliminated the wedge-failure mode.  Documented as a
v1-acceptable corner per Rule 9, with the precise condition that
would force v2 work: when Poisson-disk node placement lands as a
follow-up bead, lift `Re_HI` to 8 and verify FFW's full window.

### The Stage-1 / Stage-2 walker target list

B1's `figures/ffw2017_fig_6.jl` uses `vcat(stage1_targets,
eval_lattice)` as the walker target list, which works at its
small domain.  Carrying that pattern to Fig 1 caused immediate
walker failures: with the eval lattice at ~57k cells, the walker
tries to navigate to every cell, including ~16k cells that fall
directly on FFW's spiral arms.  The walker's 5-direction wedge
fails when surrounded by poles.

The fix is to walk to Stage-1 only and evaluate Stage-2 manually
via the visited tree's `visited_z`, `visited_pade`, `visited_h`.
This is what `path_network_solve` does internally at
`PathNetwork.jl:495-506` — we mirror that logic in the figure
script.  ~85% Stage-2 coverage (the remaining 15% are at strip
boundaries, rendered as transparent).

### The loose-vs-tight oracle's gap

The natural error-oracle pattern from worklog 036 / `FF6.1.3` is
"figure tolerance vs tighter reference tolerance".  Tried it
first for FF1.  Result: the loose run and the tight run produce
**different walker trees** (different controller-accepted step
magnitudes), so the Stage-2 nearest-visited lookup at the same
ζ-point can land on totally different visited nodes — and for
sheet-1 mid-strip samples (`0.5 + 3π i`) the lookup discrepancy
gave `|Δ| = 0.5` (not a tolerance issue — a structural one).

For sheet 0 (close to the IC) the lookup discrepancy is small
and the loose-vs-tight pin works well.  For sheet ±1 it doesn't.
Solution: pin sheet 0 with loose-vs-tight + absolute baseline +
**FFW md:122 conjugate-symmetry method** (which uses ONE walker
tree, eliminating the cross-run divergence), and pin sheet ±1
qualitatively (FF1.1.5 / FF1.1.6 structural counts) only.  This
is FFW's own methodology and gives an honest, principled error
number directly comparable to FFW Table 2.

This is a useful diagnostic worth recording: the loose-vs-tight
oracle is a **numerical-discipline** detector (does the controller
converge with tighter tol?); the FFW symmetry oracle is a
**physical-correctness** detector (does the solution have the
analytic symmetry it should?).  For complex-tree solvers where
the Stage-1 tree is sensitive to the controller setting, the
symmetry oracle is more robust.

## Hard-won lesson

**Mutation classes are orthogonal; the test must cover them all.**

M3 (fixed `h`, no `node_separation`) was expected to bite FF1.1.3
(absolute pin) because the worklog-036 narrative says "fixed `h`
fails in pole-dense regions".  Empirically — on PIII Fig 1's
specific sheet-0 sample at `0.5 + 1.0i` — M3 does NOT bite the
absolute pin: that sample is so close to the IC (`ζ = 0`, just
`|Δ| = 1.12`) that even a fixed-`h = 0.5` walker reaches it
accurately in two steps.

What M3 DOES break is the **visited tree count** (FF1.1.2):
~820 nodes vs the baseline ~2664.  Different assertion class,
different detector.

The general rule: every load-bearing mutation should bite at
least one assertion, but it does NOT have to bite the same one
the worklog-narrative predicts.  Mutation testing is empirical;
the test footer should document WHICH assertion each mutation
actually bites, not which one we expected it to.  A surprised
absence of a bite (M3 not breaking FF1.1.3) is a signal that our
test set has a useful **orthogonality** property — different
mutations break different assertions, so the full test set
catches more failure modes than any single assertion would.

The FFW Fig 6 worklog (worklog 036) hinted at this with the
distinction "convergence pin (M2) vs absolute pin (M1)".  Fig 1
extends it: FF1.1.2 (structural / count) vs FF1.1.3 (numerical /
value).  A full test set needs both classes.

## What is NOT shipped

  - **Full FFW Re_HI = 8 window**.  Reduced to `Re_HI = 5` because
    the rectangular target-grid walker hits FFW's pole-density
    wall at `Re ζ ≳ 5.5`.  Visual spiral signature is fully visible
    at `Re_HI = 5` (|z| up to ~12.2); FFW's `Re_HI = 8` extends to
    `|z| ≈ 55`, a larger zoom-out that requires Poisson-disk node
    placement.

  - **Sheet ±1 absolute-value pins**.  The Stage-2 nearest-visited
    lookup at `Im ζ ≈ ±3π` (mid sheet ±1) is sensitive to which
    visited node lands closest, and different controller tolerances
    produce different trees — making absolute pins unreliable at
    sheet ±1.  Pinned qualitatively via FF1.1.5 / FF1.1.6
    (structure / density) only.  The conjugate-symmetry method
    (FFW md:122) on sheet ±1 pairs would also work but requires
    both `Im ζ = +3π` and `Im ζ = -3π` interior to the visited
    tree, which is empirically tight at the strip boundaries —
    deferred to a follow-up bead.

  - **Pole-residue verification against FFW Table 1**.  FFW md:107
    Table 1 specifies pole residues at first-order poles
    (`c_{-1} = ±1/√γ = ±1` for `γ = 1`).  We extract pole
    locations but do not verify residue magnitudes; this would
    require integrating `u(z)·(z - z_pole)` along a small contour
    around each pole.  Deferred — the pole-count gradient
    (FF1.1.6) is the load-bearing structural check.
